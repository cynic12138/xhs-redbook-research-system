# 内容创作台完整规划

## 目标

内容创作台把当前的小红书数据抓取、笔记库、评论洞察、爆款拆解、AI 工作流和 AI 助手串成一个可执行闭环：

1. 基于关键词任务沉淀用户痛点、爆款结构和竞品表达。
2. 基于产品规则库生成小红书笔记草稿。
3. 基于规则库和平台风险做 AI 审稿。
4. 将草稿、审稿报告和修改稿保存为现有 AI 产物，进入 AI 工作台复盘。

它不是一个孤立 Prompt 面板，也不做自动发布、自动评论、自动点赞等写入动作。

## 用户需求

- 运营负责人需要批量审顾客、达人或素人写的种草笔记，识别广告腔、夸大功效、敏感医疗表达、格式不统一和人设不真实。
- 内容团队需要用结构化 Brief 快速生成小红书笔记，而不是每次手写复杂 Prompt。
- 品牌方需要把特定产品的可说卖点、禁说卖点、替代表达、场景、人设、标签沉淀为可复用规则。
- 使用者需要能从现有任务、笔记库、评论洞察和爆款模板一键进入创作或审稿。

## 成功标准

- 前端新增“内容创作台”，与现有导航、AI 助手、AI 工作台保持同一体验。
- 后端有可持久化的内容规则库、草稿、审稿记录。
- AI 审稿必须输出风险等级、问题清单、修改建议、修改稿和二次规则复核。
- 自动撰写必须基于结构化 Brief，并可引用当前任务的热门笔记、评论需求和爆款模板。
- AI 助手可以通过自然语言触发内容创作或审稿，并把结果保存为 AI 产物。
- 没有模型或 API Key 时，功能仍能返回本地规则版结果。

## 功能范围

### 内容规则库

- 产品名、行业、可说卖点、禁说卖点。
- 禁用词、敏感功效词、绝对化表达、替代表达。
- 人设、场景、语气、标签、固定输出格式。

### AI 审稿员

- 单篇审稿。
- 批量审稿留作后续扩展，第一版数据结构预留但 UI 先不做复杂批量队列。
- 输出风险等级、分数、命中问题、修改稿、标签建议。
- 默认保留原稿叙事，只做最小必要修改。

### 自动撰写小红书笔记

- 输入身份、痛点、场景、了解渠道、卖点、语气、长度、关键词。
- 生成标题、正文和标签。
- 生成后自动进入同一套审稿流程。

### AI 助手工具调用

- 工具名包括读取规则库、规则扫描、生成草稿、审稿、保存产物。
- 工具调用必须保持本地数据和读写边界，不读取 Cookie、密钥或生产数据。
- 工具失败时回退到确定性规则流程。

## 数据模型

- `ContentPlaybook`：产品规则库。
- `ContentBrief`：创作 Brief。
- `ContentDraft`：笔记草稿。
- `ContentReviewRun`：审稿记录。
- `ContentReviewIssue`：审稿问题。

所有记录使用本地 JSON 存储，与现有 `LocalStore` 模式保持一致。

## AI 工作流

新增两个工作流：

- `draft-review`：AI 审稿员。
- `note-writing`：小红书笔记撰写。

工作流产物继续写入 `aiArtifacts`，因此 AI 工作台、导出和提示词可以复用现有能力。

## 前端入口

- 侧边栏新增“内容创作台”。
- AI 助手快捷动作显示内容工作流。
- AI 工作台展示审稿和草稿产物。
- 笔记库、爆款拆解和受众洞察后续可增加“带入内容创作台”的轻入口。

## 阶段计划

1. `D-001`：数据契约、规则库、规则扫描、后端 API。
2. `D-002`：AI 审稿员单篇闭环，保存 Artifact。
3. `D-003`：结构化 Brief 撰写笔记，生成后自动审稿。
4. `D-004`：AI 助手工具调用和内容指令路由。
5. `D-005`：前端内容创作台深度适配。
6. `D-006`：测试、类型检查、桌面/移动端验证。

## 风险和边界

- 审稿不是法律意见，只是运营风险提示。
- 不自动发布、不自动评论、不自动私信。
- 若缺少真实样本数据，系统必须标明“样本不足”。
- AI 改稿必须保留人设与真实体验感，不能把素人稿改成硬广。

## 周十五蜂蜜露专项审稿闭环

`R-001`、`F-001`、`F-002` 已在现有内容创作台链路内实现，不新增平行审稿服务或 API：

1. `src/shared/contentReviewPolicy.ts` 是专项规则的单一事实来源，提示词中心、后端默认 Playbook 和前端专项模板共同引用。
2. 未绑定已保存规则时，单篇审稿、批量审稿和自动写稿后的审稿统一使用“周十五蜂蜜露专项审稿规则”，不会隐式选取第一条已保存规则。
3. 后端依次执行首次本地扫描、AI JSON 改稿或本地降级、产品名与标签确定性规范化、同规则二次扫描，并将每次结果分别保存为审稿记录和 AI Artifact。
4. 产品名统一纠正为“周十五蜂蜜露”。修改稿标签只允许产品错字纠正，数量、顺序、重复和其他原标签内容保持不变；无标题或无标签时只报告缺失，不自动虚构。
5. `draft-review` 的 builtin、guided 默认值和 advanced 默认模板共享专项政策与独立版本；其他工作流版本保持不变。内容创作台继续使用固定 `{title, body, tags}` JSON 合同，不执行用户自定义 guided/advanced 模板。
6. 内容创作台展示当前规则、规则词数和 AI/本地执行状态，可跳转到 `draft-review` 提示词，并可把任一修改稿载入表单进行人工调整和再次审稿。人工回填不会把稿件标记为定稿；自动生成草稿仍通过“接受修改稿”完成定稿。
7. 每次审稿 Artifact 包含可直接复制的 `标题：`、`内容：` 和文末原标签区块。审稿结果仅作运营风险提示，发布前仍需人工确认。

## AI 产物导航稳定性

`BUG-001` 在现有 AI 工作台和助手运行链路内修复产物被后台任务切换后无法定位的问题，不新增 API 或存储结构：

1. AI 工作台默认加载全部产物和报告，用户可显式切换为仅看当前任务；没有有效任务时回退到全部产物。
2. 内容审稿、提示词中心、Goal 和编排卡片通过现有按 ID 查询接口打开产物。即使产物不在当前任务筛选结果中，阅读区也保留已读取快照并提供返回全部产物的入口。
3. Goal 和编排的后台轮询只刷新运行记录和数据，不再自动切换当前任务、当前模块、当前产物或报告；完成结果由用户通过现有卡片按钮显式打开。
