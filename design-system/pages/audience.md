# 受众洞察页设计规则

> 页面 key：`audience`。用于从评论和互动行为里提炼用户画像、痛点和内容机会。

## 1. 页面目标

受众洞察页要帮助运营判断：

- 用户是谁。
- 用户在评论里真正问什么、怕什么、想解决什么。
- 哪些需求可以变成下一篇内容。
- 评论运营应该怎么人工回复。

## 2. 推荐布局

- 左侧：需求信号与评论主题。
- 右侧：AI 人群画像与 Markdown 洞察报告。
- 右侧报告容器应扩大为主阅读区，不要挤成窄栏。
- 左侧不是简单堆标签，而是“指标 -> 主题聚类 -> 用户原话 -> 内容机会”的层级。

## 3. 需求信号面板

- 顶部指标：样本笔记、总评论、高频主题、问题评论比例。
- 主题聚类以卡片/表格展示：主题、代表评论、情绪强度、可转化内容方向。
- 标签只用于快速扫描，不能成为主体布局。
- 评论原话要保留用户语言，但避免暴露隐私。

## 4. AI 人群画像

- 报告结构固定：受众画像摘要、评论需求聚类、用户原话素材库、未满足需求、内容机会、评论运营建议、数据缺口。
- Markdown 阅读器内部滚动。
- 没有评论样本时，报告入口禁用并说明“需要先抓取评论”。

## 5. 内容机会闭环

- 每个需求主题应提供一个“生成选题”或“加入内容策划”的入口。
- 入口使用内容策划 Prompt，不直接跳到评论发送。
- 评论回复只能生成草稿，不自动发送。

## 6. 禁用设计

- 禁止只显示一排粉色标签后留白。
- 禁止把低频评论过度解释成强需求。
- 禁止 AI 报告没有数据依据。
